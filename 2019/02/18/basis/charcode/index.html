本文简述字符、编码的相关内容，以及es6中字符相关的改变

unicode的前世今生

ASCII

ASCII是American Standard Code for Information Interchange的缩写

计算机最初起源于美国，计算机发明后有一个问题，如何在计算机中表示现实世界中的文字。
英语中的所有字符都可以用一套字母表表示，大小写一起也不过52个不同的字母，一个字节可以表示256个不同的字符，所以用一个字节表示不同的字符完全足够。
所以ASCII用一个字节表示一个字符，实际上只使用了前128位，包括大小写字母、数字和一些控制字符。

gbk等等

随着计算机的推广，其它国家也开始使用计算机，但是如何在计算机中表示本国的文字。有些国家使用了ASCII中没有使用的后128位，对于中国这样的国家，256的字符完全不够用，于是发明了自己的字符集。gbk/gb2312等都是这样的产物。字符集的演变可以参考程序员趣味读物：谈谈Unicode编码
世界各国各自使用自己的字符集，导致各国的系统无法识别他国的文件，对于互联网而言，更是不可接受。于是unicode出现了。

unicode

unicode是一套包含全世界所有文字的字符集，最大长度32位，即4个字节。
前两个字节用于表示不同的平面，后两个字节表示一个平面内的字符。每一个平面最多包含2**16=65535个字符。
unicode有17个平面（为什么只有17个平面，关于Unicode有提到)，已经使用的有7个平面，其余10个平面尚未分配任何字符。
17个平面其中一个是基本平面(BMP)，其余是辅助平面。基本平面的前两个字节都是0。所以基本平面的字符都可以用两个字节表示，有时候我们会觉得unicode是双字节的，其实不是，只是常用的字符都是双字节的。

utf16

设计unicode是一件很麻烦，复杂的工作。复杂在哪儿可以参考The Absolute Minimum Every Software Developer Absolutely, Positively Must Know About Unicode and Character Sets

所以unicode的发布是逐步的，unicode最初发布的字符集长度都不超过两个字节，所以unicode最初的字符编码ucs-2使用两个字节来表示所有字符。java发明之初，选择了ucs-2字符集。这为后续的字符问题埋下了伏笔。

在发现两个字节无法表示全世界所有字符后，unicode使用了基本平面和辅助平面,unicode的长度变为四个字节。此时usc-2已无法表示全部的unicode字符。于是出现了ucs-4和utf16。ucs-4使用四个字节表示每一个字符，只能呵呵了。utf16选择了变长，对于基本平面的字符，使用两个字节表示，对于辅助平面的字符，使用四个字节表示。这种选择，有多大程度是为了兼容ucs-2，不得而知。

java在之后选择升级到了utf16，但问题已然存在。javascript一开始就是从java半抄袭而来，ucs-2的问题自然也存在。

utf8

utf16的问题在于，对于使用英语的国家而言，之前使用ASCII只需要1M的文件，现在需要2M，这在很大程度上是不可接受的。
于是有了utf8。关于utf8的优势，可以参考关于Unicode

utf8的编码规则只有两条

  1. 单字节字符，字节第一位为0，后面7位是符号的unicode编码，所以前128位utf8和ASCII是一样的
  2. 对于n字节字符(n>=2)，第一个字节的前n位都是1，第n+1位为0，后面的字节前两位都是10,剩下的位数，从最后一位开始表示这个字符的unicode编码，剩余的位置补0。

对应这两条规则，可以使用1-6个字节表示不同的字符，实际中由于unicode现在只有17个平面，所以最多四个字节就可以表示所有字符，对应关系如下

unicode符号范围（16进制） utf8表示（2进制）
0000 0000 - 0000 007f 0xxxxxxx
0000 0080 - 0000 07ff 110xxxxx 10xxxxxx
0000 0800 - 0000 ffff 1110xxxx 10xxxxxx 10xxxxxx
0001 0000 - 001f ffff 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

utf8前128位编码和ASCII一样，对于使用英语的国家而言，utf8和ASCII看起来完全一样。
于是,utf8一统天下。但java已无法转向utf8。

es6中的字符

由于使用ucs-2字符编码，es6之前，js中的所有字符串操作都是基于双字节的。所以在处理辅助平面中的字符时，会出现问题，具体问题包括

  1. 字符串的length属性错误，对于大于0xffff的字符，长度是2
  2. 字符串的fromCharCode不能生成正确字符，实际上只会取16进制表示的最后四位
  3. 字符串的charCodeAt取得的字符编码错误
  4. 正则表达式的.,\S不能匹配编码大于0xFFFF的字符
  5. 使用index不能正确获取位于第n位的字符

es6加强了对unicode字符的支持，实际上应该就是升级到了utf-16,但是原有的api已无法改变，所以通过新增方法和属性解决了上述问题。

  1. 字符串的length属性没变，但是可以通过其他方式计算长度
  2. 字符串的fromCodePoint方法可以正确生成大于0xffff的字符
  3. 字符串的codePointAt方法可以正确获取大于0xffff的字符编码
  4. 正则表达式增加u修饰符后，.,\S可以匹配大于0xffff的字符
  5. 用for...of方法可以正确遍历字符串
