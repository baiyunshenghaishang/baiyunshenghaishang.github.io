[TOC]

Set

Set 是 es6 定义的一种新的数据结构，概念上与数组很类似，但是操作上与数组截然不同。

Set 最重要的特性是不包含重复元素，如果不需要这种特性，一般没有使用 Set 的需要。Set 内部使用Object.is来比较两个元素是否相等，而不是===,Object.is和===的区别在于NaN和自己相等以及-0和+0不相等。

创建 Set

创建 Set 只能使用构造函数的方式new Set()。也可以传入iterable对象，Set 会将迭代对象去重。如果传入的内容不能迭代，会抛出错误。

操作 Set

Set不是数组，不能够通过索引直接获取某个位置的值。
操作数组的方法包括

方法 说明
add() 添加值
delete() 删除值
has() 是否包含某个值
clear() 清空 Set
size Set 的值个数

Set 还包含forEach方法。但是由于 Set 没有索引，forEach的key和value相同，所以几乎没有任何理由使用forEach。同理，set.keys()/set.values()/set.entries()也毫无意义。遍历 Set 最好的方式是使用for...of

可以发现，Set 唯一的作用是否就是确认是否包含某个变量，但是这个功能数组也能轻易做到，arr.includes()和set.has()功能相同。
ps:和arr.indexOf()有一点区别,indexOf使用===判别是不是同一个元素,includes使用Object.is()，所以数组的includes和indexOf还是有一点区别的

Set 唯一的作用大概是用来去重

除此之外，似乎没有其他用途了

WeakSet

WeakSet 正如其名，与 Set 基本相同，区别有两点

  1. WeakSet 只能包含对象，不能包含基本类型，给 WeakSet 添加基本类型会报错
  2. WeakSet 不是可遍历对象，不包含clear()和size方法/属性

所以 WeakSet 只有三个方法add/has/delete

WeakSet 最大的特点是只会保存持有对象的弱引用。即如果 WeakSet 持有的对象不被其他变量引用，则该对象可以被回收。
WeakSet 有什么用呢，暂不明确。

Map

Map 和 Set 经常被一起提及，但是两者其实没什么关联。除了语法上的类似外，唯一是关系大概是都是 es6 添加的新数据结构。

如果说 Set 是参照数组创建的，Map 是参照对象创建的。Map也是键值对的组合，与对象的区别在于对象的key只能是字符串，而Map的key可以是任意数据类型

创建 Map

创建 Map 也只能通过new Map()的方式。如果要给Map传入初始值，可以传入每个元素都是数组的数组

操作Map

Map不能通过属性名.或者[]语法直接访问属性。Map的操作方法和Set及其相似。

方法 说明
set(key,val) 添加/更新key的值为val
get(key) 获取key的val
delete(key) 删除key
has(key) 是否包含key
clear() 清空 Map
size Map的键值对数量

除了add方法变成了set/get，其它的方法完全一样。Map同样是iterable对象，支持iterable的所有遍历方式。

Map最大的作用在于key可以不是字符串，在需要使用非字符串key时，如Symbol，Map还是很有作用的。

WeakMap

WeakMap是Map的弱引用版本，与WeakSet类似

  1. WeakMap的key只能是对象
  2. WeakMap不是可遍历对象

WeakMap的key也是弱引用，即如果WeakMap的key不再被其它变量引用，WeakMap的key可以被回收。
WeakMap的value是强引用，即只要WeakMap的key存在，WeakMap的value就不会被回收。

与WeakSet类似，WeakMap只支持set/get/delete/has四个方法

WeakMap一个很常用的应用场景是给页面上的元素添加对应值。

在不使用Map的情况下，如果元素#xiaogou被删除，对象{id:'xiaogou'}很难被回收，只能在元素被移除时通过事件触发来处理，而且这种处理也不够准确，因为元素仍然可能被重新添加会DOM中。而WeakMap几乎可以完美的解决这个问题。

还有类似场景是私有变量的处理。具体代码就不列出了。
WeakMap的使用场景一般在需要为对象保存关联信息，对象删除时可以自动删除关联信息。

Symbol.toStringTag

js中一直有一个识别类型的方法Object.prototype.toString.call(obj)，返回值[object XXX]，对于不同类型的对象,XXX各有不同。对于Array是Array，对于自定义对象，是Object。而XXX可以通过Symbol.toStringTag修改。

没错,Symbol.toStringTag的值是一个字符串，不是函数

这4个方法都接受字符串作为参数。
实际上，完全可以为一个对象定义这四个方法，然后当做正则表达式使用

动画

发表于 2019-03-14

js 动画和 css 动画的区别
GPU 加速和硬件加速是不是同一个东西
willchange 有何用，属性不同的值有什么区别

硬件加速

所谓的硬件加速，在浏览器中来讲就是使用 GPU 渲染，在 compositor thread 中直接渲染，不通过 main thread，不触发 relayout 和 repaint。

GPU 渲染会生成一个单独的composite layer，然后重排像素的位置，而不是重新绘制，所以如果使用了不能通过重排像素实现的属性，不如background，GPU 渲染带来的优势就会消失

会生成单独composite layer的情况包括

  1. transform 3d 或者 perspective
  2. video or canvas
  3. 使用 css filters
  4. 一个覆盖在其它composite layer上的元素

通过 chrome 开发者工具more tools中的layers可以查看页面的composite layer

不是所有属性都会直接在composite layer中完成，只有以下属性会

  1. transform
  2. opacity
  3. filter

注意：opacity 虽然会在 GPU 中直接完成，但是在改变之前并不会生成composite layer。而transform 3d才会直接生成composite layer，但是所有的transform变化都会在 GPU 中直接完成

所以，如果是transform 2d动画，在动画的开始和结束会各有一次repaint

如果在动画开始前就给元素设置transform:translateZ(0)，会直接生成composite layer，动画过程中不会有任何repaint
但是transofrm:translateZ(0)或者其他类似方式毕竟是 hack 类型的，所以 css 提出了新的属性will-change
will-change应该不仅仅创建一个composite layer，但从动画的角度来看，可以这样认为。
will-change的使用有几点需要注意：

  1. 尽量不要使用。和硬件加速一样，会有额外的内存消耗，尤其在移动设备上。
  2. 使用完成后需要移除。
  3. 在属性变化之前加上，否则毫无意义。
  4. 不要给 will-change 添加太多属性，易燃易爆炸

什么是定型数组

JS中数组一直是不限类型的，一个数组中可以同时存在多种不同类型的数据。普通的数组算术运算速度无法满足WebGL的需求，所以es6引入了定型数组。
定型数组是用于处理数值类型的(不是所有类型)的专用数组，定型数组不是数组类型。

定型数组类型

不同于普通的数组，定型数组的长度是固定的，创建之后就不能改变。所以定型数组可以在内存中对应一段固定的地址，这就是数组缓冲区。

不能通过数组缓冲区直接修改数据，要修改缓冲区的数据，必须创建视图。

对对象遍历以及相关知识的梳理

对象的属性

对象的属性分很多种，obj.a获取只是获取一个值，但是属性的来源可能有多种。与对象遍历相关的有4种

  1. 是否是继承的属性
  2. 是否可以枚举，即属性的enumerable是否为true
  3. 是数据属性还是访问器属性
  4. 是否是Symbol属性

属性遍历

obj.a方式获取的属性包括以上所有类型的属性，但是属性遍历时，除非有特殊需求，一般只考虑自身属性

继承属性

能够遍历继承属性的方式只有一种：for...in
for...in会遍历自身和继承的可枚举属性

这是唯一一个获取继承属性的方法，其它遍历属性的方法，都只能获取自身的属性。
由于for...in会遍历继承的属性，所以在使用for...in遍历对象属性时，一般需要加上hasOwnProperty来判断是否是对象本身的属性

还有一种方法是使用Object.keys()

Object.keys()

自身可枚举属性应该是对象属性遍历时最常用的情况。Object.keys()返回对象自身可枚举属性的数组

上面代码和继承属性中for...in的遍历效果一样，都是遍历自身可枚举属性。
所以推荐使用Object.keys()遍历对象属性，而不是for...in，因为obj.hasOwnProperty()实在容易忘记

有一点需要注意，Object.keys()只有IE9+支持，但for...in支持度至少到IE8，具体不清楚。如果要兼容低版本IE，则只能使用for...in。

不可枚举属性

如果要遍历对象的不可枚举属性，需要使用Object.getOwnPropertyNames()。此方法会返回自身可枚举和不可枚举的属性的数组。

Symbol属性

理论上来讲，Symbol属性可以是可枚举的，也可以是不可枚举的。因为一般判断属性是否可以枚举是通过obj.propertyIsEnumerable()或者Object.getOwnPropertyDescriptor()返回对象的enumerable属性。
虽然Symbol属性的enumerable属性与其它属性一样，但是Symbol属性不会被前面任何一种方法返回。

虽然上述sy和age属性的enumerable都为true，但是只有age属性会被Object.keys()返回，换成for...in或者Object.getOwnPropertyNames()也是一样。

如果要遍历Symbol属性，需要使用Object.getOwnPropertySymbols(),此方法会返回自身的Symbol属性的数组，注意，只有Symbol属性。

自身所有属性

最后一种遍历对象属性的方法是Reflect.ownKeys(),此方法会返回对象的所有属性的数组，包括Symbol和非Symbol属性，枚举和不可枚举属性。可以认为
Reflect.ownKeys = Object.getOwnPropertyNames() + Object.getOwnPropertySymbols()

总结

上面讲了5中对象属性遍历的方法,可以用表格总结如下

方法 特性
Reflect.ownKeys() 所有属性，包括Symbol和非Symbol属性
Object.getOwnPropertySymbols() Symbol属性
Object.getOwnPropertyNames() 非Symbol属性，包括可枚举属性和不可枚举属性
Object.keys() 非Symbol的可枚举属性
for…in 非Symbol的自身和继承的可枚举属性

一般情况下，不建议使用for...in，因为for...in属性的来源很难判断，可能来自自身，也可能来自继承的对象，甚至继承对象的继承对象。如果要获取继承对象的可枚举属性，最好使用Object.keys(Object.getPrototypeOf(obj))。

其它涉及对象遍历的方法

有一些方法，虽然不直接遍历对象的属性，但是会隐含有对象属性遍历的操作，以下做一些梳理，可能有遗漏

JSON.stringify()

序列化自身可枚举属性,不包含Symbol属性,对于访问器属性，会使用get的返回值

Object.assign() / …

复制对象的自身可枚举属性，包含Symbol属性，对于访问器属性，会使用get的返回值

Object.assign()和JSON.stringify()在属性遍历上的区别只在于Object.assign()会包含Symbol属性

...和Object.assign的属性遍历行为完全一样

Object.getOwnPropertyDescriptors()

返回自身所有属性的描述符，遍历行为等同于Reflect.ownKeys()。

obj.hasOwnProperty

检测对象自身是否包含某个属性，检测自身所有属性，不包含继承属性

in 运算符

in运算符遍历自身和继承的所有属性,包括Symbol属性

字符、编码

发表于 2019-02-18

[TOC]

本文简述字符、编码的相关内容，以及es6中字符相关的改变

unicode的前世今生

ASCII

ASCII是American Standard Code for Information Interchange的缩写

计算机最初起源于美国，计算机发明后有一个问题，如何在计算机中表示英文字母。

由于英语中的所有的单词都可以用大小写字母表示，而大小写字母和数字加在一起也不过62个字符。而当时的计算机普遍使用8个比特来表示一个字节(byte),一个字节可以表示256个不同的字符，所以用一个字节表示不同的字符完全足够。

所以ASCII用一个字节表示一个字符。实际上只使用了前128位，包括大小写字母、数字和一些控制字符，后128位，从128到255没有定义任何字符。
到此为止，一切都很完美。

gbk…

随着计算机的推广，其它国家也开始使用计算机，他们马上也遇到了同样的问题，如何在计算机表示本国的文字。

很快大家就想到了使用ASCII中128-255的位置来表示本国字符。于是产生一大批不同的字符集，他们的前128位都是相同的，但是后128位就自己玩自己的了。如果你从法国发一封电子邮件到德国，使用了后128位中的字符，那么到德国的时候，他们就可能都变成了“????”。

计算机进入亚洲之后，事情变得更加复杂。而对于中国这样的国家，汉字的数量成千上万，256的字符完全不够用。于是我们使用两个字节来表示一个汉字。但是前128位还是和ASCII一样，所以这些字符集表示英文是一个字节，表示中文是两个字节。gb2312、gbk、gb18030都是这样的产物，区别只在于后面的字符集比前面的字符集包含的汉字更多，而且都是向下兼容的。

如果说在国内使用同一种字符集还可以接受，毕竟大部分人跨国发邮件的机会并不多。但是互联网出现之后，这个问题变得格外重要，毕竟你总不能在电脑上把世界上所有的字符集都装上吧，而且一旦文件没有指定字符集，能够正确打开文件的机会几乎为0。为了解决这个问题，unicode横空出世。

unicode

互联网的出现让设计一套包含全世界所有字符的字符集的任务迫在眉睫，有两个国际组织各自设计了一套，一个是unicode组织，一个是ucs组织。很明显，这个世界不需要两套统一字符集，所以他们发现对方的存在后很快达成了一致，合并。
1991年，两个团队决定合并字符集。从此以后，两个组织仍然独立公布各自的标准，但是两者的编码完全兼容，并共同调整未来的扩展。

unicode是一套包含全世界所有字符的字符集，长度4个字节。

前两个字节用于表示不同的平面，后两个字节表示一个平面内的字符。每一个平面最多包含2**16=65535个字符。
为什么unicode只有17个平面，可以参考这篇文章

17个平面其中一个是基本平面(BMP)，其余是辅助平面,现在已使用的辅助平面有6个。基本平面的前两个字节都是0。所以基本平面的字符都可以用两个字节表示。

ucs-2/utf16

由于设计这样一套字符是非常复杂、麻烦的工作，所以unicode是分期发布的。每完成一部分的工作就会发布一部分。

unicode最初发布的字符集字符长度只有两个字节，所以unicode最初的字符编码ucs-2使用两个字节来表示所有字符。javascript发明之初，选择了ucs-2字符集。这为后续的字符问题埋下了伏笔。实际上，unicode提出辅助平面在javascript发明之后，而实际公布辅助平面字符要等到2001年，unicode3.1。

在发现两个字节无法表示全世界所有字符后，unicode使用了基本平面和辅助平面,unicode的长度变为四个字节。此时usc-2已无法表示全部的unicode字符。于是出现了ucs-4和utf16。ucs-4使用四个字节表示每一个字符。utf16选择了不固定长度，对于基本平面的字符，使用两个字节表示，对于辅助平面的字符，使用四个字节表示。

utf8

utf16的问题在于，对于使用英语的国家而言，之前使用ASCII只需要1M的文件，现在需要2M，这在很大程度上是不可接受的。而utf8很好的解决了这个问题。

utf8的编码规则只有两条

  1. 单字节字符，字节第一位为0，后面7位是符号的unicode编码，所以前128位utf8和ASCII是一样的
  2. 对于n字节字符(n>=2)，第一个字节的前n位都是1，第n+1位为0，后面的字节前两位都是10,剩下的位数，从最后一位开始表示这个字符的unicode编码，剩余的位置补0。

对应这两条规则，可以使用1-6个字节表示不同的字符，实际中由于unicode现在只有17个平面，所以最多四个字节就可以表示所有字符，对应关系如下

unicode符号范围（16进制） utf8表示（2进制）
0000 0000 - 0000 007f 0xxxxxxx
0000 0080 - 0000 07ff 110xxxxx 10xxxxxx
0000 0800 - 0000 ffff 1110xxxx 10xxxxxx 10xxxxxx
0001 0000 - 001f ffff 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

utf8前128位编码和ASCII一样，对于使用英语的国家而言，utf8和ASCII看起来完全一样。
于是,utf8一统天下。
但其实对于中文而言,utf8的效率并不高，中文字符在utf8中一般使用三个字节表示。

字符集与字符编码

字符集和字符编码是两个很相近的概念，很多时候也可以互换使用。对于unicode之外的字符集，我们基本可以认为字符集和字符编码是一个概念。因为对于这些字符集而言，只有一套字符编码与他对应。

但是unicode不同，unicode这套字符集有多套与其对应的字符编码，如ucs-2、ucs-4、utf16、utf8。对于这些字符编码而言，区分字符集和字符编码就比较重要了。

最重要的是理解在字符编码中的编码是0xabcdabcd，并不代表在unicode字符集中的编码(准确来讲叫码点)也是0xabcdabcd。实际上我们一般也不在乎在字符编码中的编码是多少，一般所说的编码都是字符在unicode中对应的编码。

js中的字符问题

对于unicode中字符编码大于0xffff的字符，其对应的unicode编码有四个字节，而js使用的ucs-2编码，认为所有的字符都是两个字节，所以会将四个字节的unicode字符当做两个字符。

此处有一点要注意，unicode0x20BB7的编码被当成的两个字符的编码并不是0x0002和0xBB7。而是\uD842和\uDFB7。\uD842\uDFB7其实是编码0x20BB7在utf16中的表示方法，关于如何将unicode编码转换为utf16，看这里。

而为什么unicode0x20BB7的编码是以utf16的形式呈现，而不是utf8,ucs-4,utf32等等，不知道。猜测可能是以前使用ucs-2将字符转换成二进制编码，但是后来为了识别unicode编码大于0xffff的字符，升级到兼容ucs-2的utf16。

js中字符的具体问题包括但不限于以下

  1. 字符串的length属性错误，对于大于0xffff的字符，长度是2
  2. 字符串的fromCharCode不能生成正确字符，实际上只会取16进制表示的最后四位，即最后两个字节
  3. 字符串的charCodeAt取得的字符编码错误。实际上大部分字符串的方法都有问题
  4. 正则表达式的.,\S不能匹配编码大于0xFFFF的字符
  5. 使用index不能正确获取位于第n位的字符

es6加强了对unicode字符的支持，实际上应该就是升级到了utf-16,但是原有的api已无法改变，所以通过新增方法和属性解决了上述问题。

  1. 字符串的length属性没变，但是可以通过其他方式计算长度
  2. 字符串的fromCodePoint方法可以正确生成大于0xffff的字符
  3. 字符串的codePointAt方法可以正确获取大于0xffff的字符编码
  4. 正则表达式增加u修饰符后，.,\S可以匹配大于0xffff的字符
  5. 用for...of方法可以正确遍历字符串
