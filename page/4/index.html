本文简述字符、编码的相关内容，以及es6中字符相关的改变

unicode的前世今生

ASCII

ASCII是American Standard Code for Information Interchange的缩写

计算机最初起源于美国，计算机发明后有一个问题，如何在计算机中表示英文字母。

由于英语中的所有的单词都可以用大小写字母表示，而大小写字母和数字加在一起也不过62个字符。而当时的计算机普遍使用8个比特来表示一个字节(byte),一个字节可以表示256个不同的字符，所以用一个字节表示不同的字符完全足够。

所以ASCII用一个字节表示一个字符。实际上只使用了前128位，包括大小写字母、数字和一些控制字符，后128位，从128到255没有定义任何字符。
到此为止，一切都很完美。

gbk…

随着计算机的推广，其它国家也开始使用计算机，他们马上也遇到了同样的问题，如何在计算机表示本国的文字。

很快大家就想到了使用ASCII中128-255的位置来表示本国字符。于是产生一大批不同的字符集，他们的前128位都是相同的，但是后128位就八仙过海各显神通了。如果你从法国发一封电子邮件到德国，使用了后128位中的字符，那么到德国的时候，他们就可能都变成了“????”。

计算机进入亚洲之后，事情变得更加复杂。而对于中国这样的国家，汉字的数量成千上万，256的字符完全不够用。于是我们使用两个字节来表示一个汉字。但是前128位还是和ASCII一样，所以这些字符集表示英文是一个字节，表示中文是两个字节。gb2312、gbk、gb18030都是这样的产物，区别只在于后面的字符集比前面的字符集包含的汉字更多，而且都是向下兼容的。

如果说在国内使用同一种字符集还可以接受，毕竟大部分人跨国发邮件的机会并不多。但是互联网出现之后，这个问题变得格外重要，毕竟你总不能在电脑上把世界上所有的字符集都装上吧，而且一旦文件没有指定字符集，能够正确打开文件的机会几乎为0。为了解决这个问题，unicode横空出世。

unicode

互联网的出现让设计一套包含全世界所有字符的字符集的任务迫在眉睫，有两个国际组织各自设计了一套，一个是unicode组织，一个是ucs组织。很明显，这个世界不需要两套统一字符集，所以他们发现对方的存在后很快达成了一致，合并。
1991年，两个团队决定合并字符集。从此以后，两个组织仍然独立公布各自的标准，但是两者的编码完全兼容。

unicode是一套包含全世界所有字符的字符集，长度4个字节。

前两个字节用于表示不同的平面，后两个字节表示一个平面内的字符。每一个平面最多包含2**16=65535个字符。
为什么unicode只有17个平面，可以参考这篇文章

17个平面其中一个是基本平面(BMP)，其余是辅助平面,现在已使用的辅助平面有6个。基本平面的前两个字节都是0。所以基本平面的字符都可以用两个字节表示。

ucs-2/utf16

由于设计这样一套字符是非常复杂、麻烦的工作，所以unicode是分期发布的。每完成一部分的工作就会发布一部分。

unicode最初发布的字符集字符长度只有两个字节，所以unicode最初的字符编码ucs-2使用两个字节来表示所有字符。javascript发明之初，选择了ucs-2字符集。这为后续的字符问题埋下了伏笔。实际上，unicode提出辅助平面在javascript发明之后，而实际公布辅助平面字符要等到2001年，unicode3.1。

在发现两个字节无法表示全世界所有字符后，unicode使用了基本平面和辅助平面,unicode的长度变为四个字节。此时usc-2已无法表示全部的unicode字符。于是出现了ucs-4和utf16。ucs-4使用四个字节表示每一个字符。utf16选择了不固定长度，对于基本平面的字符，使用两个字节表示，对于辅助平面的字符，使用四个字节表示。

utf8

utf16的问题在于，对于使用英语的国家而言，之前使用ASCII只需要1M的文件，现在需要2M，这在很大程度上是不可接受的。而utf8很好的解决了这个问题。

utf8的编码规则只有两条

  1. 单字节字符，字节第一位为0，后面7位是符号的unicode编码，所以前128位utf8和ASCII是一样的
  2. 对于n字节字符(n>=2)，第一个字节的前n位都是1，第n+1位为0，后面的字节前两位都是10,剩下的位数，从最后一位开始表示这个字符的unicode编码，剩余的位置补0。

对应这两条规则，可以使用1-6个字节表示不同的字符，实际中由于unicode现在只有17个平面，所以最多四个字节就可以表示所有字符，对应关系如下

unicode符号范围（16进制） utf8表示（2进制）
0000 0000 - 0000 007f 0xxxxxxx
0000 0080 - 0000 07ff 110xxxxx 10xxxxxx
0000 0800 - 0000 ffff 1110xxxx 10xxxxxx 10xxxxxx
0001 0000 - 001f ffff 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

utf8前128位编码和ASCII一样，对于使用英语的国家而言，utf8和ASCII看起来完全一样。
于是,utf8一统天下。
但其实对于中文而言,utf8的效率并不高，中文字符在utf8中一般使用三个字节表示。

字符集与字符编码

字符集和字符编码是两个很相近的概念，很多时候也可以互换使用。对于unicode之外的字符集，我们基本可以认为字符集和字符编码是一个概念。因为对于这些字符集而言，只有一套字符编码与他对应。

但是unicode不同，unicode这套字符集有多套与其对应的字符编码，如ucs-2、ucs-4、utf16、utf8。对于这些字符编码而言，区分字符集和字符编码就比较重要了。

最重要的是理解在字符编码中的编码是0xabcdabcd，并不代表在unicode字符集中的编码(准确来讲叫码点)也是0xabcdabcd。实际上我们一般也不在乎在字符编码中的编码是多少，一般所说的编码都是字符在unicode中对应的编码。

js中的字符问题

对于unicode中字符编码大于0xffff的字符，其对应的unicode编码有四个字节，而js使用的ucs-2编码，认为所有的字符都是两个字节，所以会将四个字节的unicode字符当做两个字符。

此处有一点要注意，unicode0x20BB7的编码被当成的两个字符的编码并不是0x0002和0xBB7。而是\uD842和\uDFB7。\uD842\uDFB7其实是编码0x20BB7在utf16中的表示方法，关于如何将unicode编码转换为utf16，看这里。

而为什么unicode0x20BB7的编码是以utf16的形式呈现，而不是utf8,ucs-4,utf32等等，不知道。猜测可能是以前使用ucs-2将字符转换成二进制编码，但是后来为了识别unicode编码大于0xffff的字符，升级到兼容ucs-2的utf16。

js中字符的具体问题包括但不限于以下

  1. 字符串的length属性错误，对于大于0xffff的字符，长度是2
  2. 字符串的fromCharCode不能生成正确字符，实际上只会取16进制表示的最后四位，即最后两个字节
  3. 字符串的charCodeAt取得的字符编码错误。实际上大部分字符串的方法都有问题
  4. 正则表达式的.和\S不能匹配编码大于0xFFFF的字符
  5. 使用index不能正确获取位于第n位的字符

es6加强了对unicode字符的支持，实际上应该就是升级到了utf-16,但是原有的api已无法改变，所以通过新增方法和属性解决了上述问题。

  1. 字符串的length属性没变，但是可以通过其他方式计算长度
  2. 字符串的fromCodePoint方法可以正确生成大于0xffff的字符
  3. 字符串的codePointAt方法可以正确获取大于0xffff的字符编码
  4. 正则表达式增加u修饰符后，.和\S可以匹配大于0xffff的字符
  5. 用for...of方法可以正确遍历字符串

vue-cli-preset 项目地址

为什么要用 preset

最近想搭建几个 vue 的项目，考虑到多一事不如少一事（反正我也不会脚手架），就想使用 vue-cli3 来搭建。试了下，配置项还是挺全的，但是每次新建项目都要设置很多选项，有些文件可能还不合心意。比如我想加个.prettierrc,但是 vue-cli 没有，有也可能与我的习惯不合，怎么办。

翻了下文档，看到 vue-cli 可以配置 preset，就是一个预先设置好的配置，使用这个配置创建项目，创建出来的项目就基本可以直接用了。

preset 搭建流程

官方文档在此

vue-cli 支持本地的 preset 和远程 preset，考虑到后续换电脑可能还得用这个 preset，虽然到时候可能 vue 都凉凉了，用远程 preset。

preset.json

关于preset.json的可选配置项，官方文档给出了一个例子，使用vue create创建项目后也会生成一份类似的配置,使用vue config可以查看。所有的可配置项，plugins具体如何配置，完全找不到文档，所以此处基本使用vue config后展示的配置。其它需要的配置都是通过generator.js

generator.js

generator 用于修改项目已有文件和插入新的文件。

栗子

generator.js导出一个函数，接受三个参数，api,options,rootOptions

api

用于生成和修改文件，主要有三个方法

  1. render 复制文件，接受字符串或对象，如果是字符串，复制到根目录，如果是对象，key 表示要复制到的目录，value 表示文件路径，
  2. extendPackage 用于修改package.json字段，会合并到同名字段
  3. postProcessFiles 用于修改已有文件，写法参见项目的changebrowserslist.js

复制文件尽量使用目录，而不是文件名，文件名以.开头的，需要改成_，官方文档有详细介绍
